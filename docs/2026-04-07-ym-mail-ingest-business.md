# YM Mail Ingest MVP

## 1. Бизнес-задача

В почтовый ящик `ya-stats@solta.io` регулярно приходят клиентские отчеты во вложениях. Сейчас эти файлы нужно вручную находить, открывать, понимать, какие из них относятся к нужным темам, и отдельно проверять, есть ли внутри рабочая таблица с UTM-структурой. Это медленно, нестабильно и плохо масштабируется.

Цель MVP: автоматизировать ежедневный intake таких вложений из почты в контролируемый технический слой, чтобы:

- не искать файлы вручную в почте;
- не засорять Google Drive аккаунта `ya-stats@solta.io`;
- сохранять только те файлы, которые реально содержат нужную табличную структуру;
- получить единый список найденных файлов для последующей обработки в Python.

На текущем этапе задача не состоит в финальной нормализации данных и не состоит в построении union-таблицы. MVP заканчивается на отборе валидных файлов и записи их метаданных в manifest.

## 2. Бизнес-контекст

### Источники

- Почтовый ящик: `ya-stats@solta.io`
- Spreadsheet с темами и техническими листами: `17izchH29LyxuTCNWJ0SThSXmuubMnNFCjtPJiWtcxFA`
- Лист с темами: `отчеты`
- Папка для staging-файлов: `1f38YjgK214y-KNH8oSqoQm8EWbZSfywn`

### Основная проблема

Вложения могут быть похожи по названию, но не все из них содержат нужную таблицу. Простое наличие слова `utm` в произвольной строке файла недостаточно, потому что оно может встречаться в названии отчета, пояснении над таблицей или служебном тексте. Поэтому валидным считается только файл, в котором найден табличный блок с шапкой, и в самой строке шапки присутствует хотя бы одна UTM-колонка.

## 3. Бизнес-результат MVP

После каждого запуска за текущий день система должна:

- просмотреть письма за сегодня в ящике `ya-stats@solta.io`;
- отобрать письма по темам из листа `отчеты`;
- проверить вложения форматов `xlsx` и `csv`;
- сохранить в staging только валидные файлы;
- записать техметаданные по найденным файлам в лист `manifest`.

Результатом MVP является не “готовый отчет”, а надежный технический intake-слой для следующей фазы.

## 4. Бизнес-правила

### 4.1. Правила отбора писем

- Поиск идет в самом ящике `ya-stats@solta.io`.
- Фильтра по отправителю нет.
- Обрабатываются только письма за текущую дату по часовому поясу скрипта.
- Темы для отбора берутся из листа `отчеты`, столбец `A`.
- Совпадение темы неполное: строка из ячейки разбивается на слова, письмо подходит, если все слова из ячейки встречаются в теме письма, даже если тема письма длиннее.

### 4.2. Правила отбора файлов

- Рассматриваются только вложения `xlsx` и `csv`.
- Файл считается валидным только если внутри найден табличный блок с шапкой.
- В строке шапки должна быть хотя бы одна из колонок:
  - `utm_source`
  - `utm_campaign`
  - `utm_content`
  - `utm_term`
- Упоминания `utm` вне строки шапки не считаются основанием для отбора.
- Если файл не содержит такой табличной структуры, он пропускается.

### 4.3. Правила хранения

- Валидные файлы сохраняются в папку `1f38YjgK214y-KNH8oSqoQm8EWbZSfywn`.
- Внутри staging-папки создается подпапка по дате запуска в формате `YYYY-MM-DD`.
- Исходные файлы на этапе MVP не удаляются и остаются как архив.
- Рабочие файлы не должны создаваться в `My Drive` аккаунта `ya-stats@solta.io`; staging хранится на другом аккаунте, который дал доступ к папке.

### 4.4. Правила фиксации в manifest

- Для каждого сохраненного файла создается запись в листе `manifest`.
- В manifest должны храниться минимум:
  - дата запуска;
  - matched topic;
  - subject письма;
  - дата письма;
  - message id;
  - attachment name;
  - drive file id;
  - статус;
  - признак того, относится ли файл к самому позднему письму по теме за день.

## 5. Бизнес-процессы

### Процесс 1. Ежедневный intake из почты

1. Запускается Apps Script в ящике `ya-stats@solta.io`.
2. Скрипт определяет “сегодня” по часовому поясу скрипта.
3. Скрипт читает список тем из листа `отчеты`.
4. Скрипт ищет письма за текущий день и отбирает те, чьи темы совпадают с правилами неполного вхождения.
5. Для каждого подходящего письма скрипт проверяет вложения `xlsx/csv`.
6. Для каждого файла скрипт определяет, есть ли внутри таблица с шапкой и UTM-колонками.
7. Валидные файлы сохраняются в staging-папку текущей даты.
8. По каждому сохраненному файлу в `manifest` создается техническая запись.
9. В `manifest` отдельно помечается, какое письмо по каждой теме является самым поздним за день.

### Процесс 2. Повторный прогон за тот же день

Текущий intake MVP ориентирован на ежедневный запуск “за сегодня”. Если запуск за тот же день повторяется, система должна заново сформировать staging/manifest для этого дня без ручного поиска писем. Точная политика очистки staging в MVP может быть технически выбрана отдельно, но business intent такой: повторный прогон не должен требовать ручной перебор почты человеком.

### Процесс 3. Следующая фаза после MVP

После того как intake стабильно собирает валидные файлы:

1. Python читает `manifest`.
2. Python берет только файлы, относящиеся к самому позднему письму по теме за день.
3. Python открывает файлы, извлекает таблицы и начинает нормализацию.
4. На следующем этапе данные загружаются в proxy-слой spreadsheet и только потом используются в отчетах.

Эта фаза не входит в текущий MVP, но именно под нее строится intake-слой.

## 6. Роли и ответственность

- `ya-stats@solta.io`
  - источник почтовых данных;
  - контур запуска Apps Script;
  - не используется как место хранения рабочих staging-файлов в `My Drive`.

- Аккаунт-владелец staging-папки и spreadsheet
  - владеет папкой `1f38YjgK214y-KNH8oSqoQm8EWbZSfywn`;
  - владеет spreadsheet `17izchH29LyxuTCNWJ0SThSXmuubMnNFCjtPJiWtcxFA`;
  - получает на своем Drive технические staging-файлы.

- Apps Script
  - выполняет intake;
  - не делает финальную бизнес-нормализацию;
  - пишет технический manifest.

- Python
  - следующая фаза;
  - читает уже отобранные файлы;
  - делает глубокий разбор структуры и последующую нормализацию.

## 7. Ограничения и осознанные решения

- На этапе MVP не строится единая union-таблица по всем структурам файлов.
- На этапе MVP не фиксируется окончательная схема будущего `proxy_raw`.
- На этапе MVP не требуется прямой доступ Python к Gmail.
- На этапе MVP не нужен фильтр по отправителю.
- На этапе MVP важнее не потерять валидный файл, чем слишком рано ужать логику под финальную модель данных.

## 8. Критерии приемки MVP

MVP считается успешным, если:

- за текущую дату система находит письма в ящике `ya-stats@solta.io` по темам из листа `отчеты`;
- в staging попадают только `xlsx/csv`, где найден табличный блок с шапкой и хотя бы одной UTM-колонкой;
- staging-файлы создаются в папке `1f38YjgK214y-KNH8oSqoQm8EWbZSfywn` по подпапкам дат;
- в spreadsheet создается и заполняется лист `manifest`;
- у каждого файла в manifest есть достаточный набор метаданных для следующей обработки в Python;
- процесс не требует ручного открытия писем для первичного отбора.
