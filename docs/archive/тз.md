ТЗ на пайплайн загрузки конверсий и постклик-метрик из выгрузок Яндекс.Метрики в рабочий spreadsheet

Контекст и ограничения

Источник данных — выгрузки Яндекс.Метрики, которые приходят на почту вложениями.
Прямого доступа к Метрике нет.
Одна тема письма соответствует одному потоку отчетов / одной рекламной кампании.
По одной теме может приходить несколько файлов и несколько писем.
Названия файлов вложений не являются стабильными и не должны использоваться в логике.
Дата письма не равна дате отчета. Дата отчета должна извлекаться из содержимого файла.
Форматы выгрузок между клиентами и агентствами неконсистентны:

* порядок столбцов разный
* часть столбцов может отсутствовать
* набор UTM и метрик может отличаться
* цели клиентов не нужно унифицировать по смыслу между собой

Цель системы

Автоматически собирать все релевантные вложения по заданным темам писем, нормализовывать табличные выгрузки, приводить их к единому каноническому набору полей и публиковать в рабочий spreadsheet финальную wide-таблицу с:

* полным superset UTM-полей
* полным superset метрик
* обезличенными целями в виде `goal_1 ... goal_N`

Proxy spreadsheet

Proxy spreadsheet используется как business-registry и хранит только:

* в колонке A — тема письма
* в колонках B:... — расшифровка обезличенных целей для этой темы

Структура:

* A1 — `Тема письма`
* B1:... — `Цель 1`, `Цель 2`, ..., `Цель N`
* A2:A — конкретные темы писем
* B2:... — реальные названия целей для соответствующей темы

Правила:

* одна строка = одна тема / один поток отчетов
* `goal_1` в одной теме не обязан совпадать по смыслу с `goal_1` в другой теме
* proxy spreadsheet является справочником тем и дешифровки целей, но не источником численных значений метрик
* если новая цель появилась в отчете, а расшифровка в proxy spreadsheet пока не заполнена, пайплайн не должен ломаться: значение грузится в соответствующий `goal_N`, а название считается пустым до заполнения вручную
* если число целей превысило текущий диапазон столбцов spreadsheet, диапазон расширяется вправо без изменения уже назначенных goal-слотов

Общая архитектура

Рекомендуемый контур:
Gmail -> Apps Script -> Supabase raw/storage -> Python normalization -> export view -> рабочий spreadsheet

Роли компонентов:

1. Apps Script

   * читает список тем из proxy spreadsheet
   * ищет письма по темам
   * выгружает вложения как есть
   * отправляет файлы и метаданные в Supabase
   * не делает сложной нормализации

2. Supabase

   * хранит реестр файлов и сырой слой
   * хранит нормализованные данные
   * хранит export-view для выгрузки в рабочий spreadsheet

3. Python backend

   * классифицирует вложения
   * нормализует файлы
   * извлекает дату отчета
   * сопоставляет заголовки с каноническими полями
   * собирает канонический слой
   * строит wide-экспорт

Логика загрузки писем и файлов

Нельзя работать по модели “письма за вчера/сегодня”.
Нужно работать по модели “инкрементальная синхронизация всех файлов по теме в скользящем окне, покрывающем типичные досылки”.

Базовое окно синхронизации для MVP:

* текущий календарный месяц
* предыдущий календарный месяц

Для каждой активной темы система должна:

* искать все письма в окне синхронизации по этой теме
* забирать все вложения
* сравнивать с уже загруженными файлами
* догружать только новые
* повторно не дублировать уже известные

Ключевые правила:

* ориентироваться на полный реестр файлов в окне синхронизации, а не на последнее письмо
* имя файла не использовать в логике
* одно и то же вложение определять по содержимому, а не по названию

Дедупликация файлов

Файл считается уже загруженным по `file_hash`.
Дополнительно хранить:

* `message_id`
* `thread_id`
* `attachment_index`
* `attachment_name`
* `message_subject`
* `message_date`

Главный критерий дедупликации — хэш файла.
Имя файла используется только для логов и дебага.

Классификация вложений

Каждое вложение должно классифицироваться как:

* `table`
* `graph`
* `unknown`
* `broken`

В основной нормализованный слой попадают только файлы типа `table`.
Файлы типа `graph` можно хранить в raw/debug-слое, но не тащить в основную факт-таблицу.

Дата отчета

Дата отчета берется из содержимого файла, а не из даты письма.

Приоритет:

1. если в строках есть поле типа `Дата визита`, использовать его как `report_date`
2. если в строках нет даты, но в шапке файла есть период отчета, извлекать `report_date_from` и `report_date_to`
3. `message_date` хранить отдельно как технический атрибут источника

Нормализация файлов

Парсинг должен выполняться по заголовкам колонок, а не по позициям столбцов.

Нельзя предполагать:

* фиксированный порядок столбцов
* обязательное наличие всех UTM
* обязательное наличие всех метрик
* обязательное наличие одних и тех же целей
* одинаковый формат у всех клиентов

Нужно:

* нормализовать заголовки
* распознавать синонимы заголовков
* строить словарь `нормализованный_заголовок -> значение`
* работать с полным superset полей

Примеры нормализации заголовков:

* `UTM Source`, `utm_source`, `Метка UTM source` -> `utm_source`
* `UTM Medium`, `utm_medium` -> `utm_medium`
* `Визиты`, `Visits` -> `visits`

Поддержка полей

Система должна поддерживать полный superset всех измерений и метрик, даже если в конкретном файле часть полей отсутствует.

Это касается:

* всех UTM-полей
* всех других dimensions
* всех метрик
* всех целей

Если поле в конкретном отчете отсутствует, в нормализованной записи оно считается пустым, но поддержка этого поля в системе сохраняется.

Внутренний канонический слой

Рекомендуется делать внутренний нормализованный слой не wide, а канонический sparse-слой.

Состав слоя:

1. `fact_rows`
   Одна строка = одна нормализованная запись из отчета.
   Поля:

* `fact_row_id`
* `topic`
* `file_id`
* `report_date`
* `report_date_from`
* `report_date_to`
* `message_date`
* `layout_signature`
* `row_hash`
* `is_current`

2. `fact_dimensions`
   Поля:

* `fact_row_id`
* `dimension_key`
* `dimension_value`

Примеры:

* `utm_source`
* `utm_medium`
* `utm_campaign`
* `utm_content`
* `utm_term`
* любые другие dimensions, встречающиеся в отчетах

3. `fact_metrics`
   Поля:

* `fact_row_id`
* `metric_key`
* `metric_value`

Примеры:

* `visits`
* `visitors`
* `clicks`
* `impressions`
* `bounce_rate`
* `page_depth`
* `avg_visit_duration_seconds`
* `goal_1 ... goal_N`

Почему так:

* не ломается от нового поля
* не требует миграции схемы при новом клиенте
* не зависит от порядка столбцов
* позволяет поддерживать любой superset измерений и метрик

Цели

Цели не нужно объединять по смыслу между клиентами.
Они должны обезличиваться в `goal_1 ... goal_N`.

Правила:

* goal-колонки из исходного файла переводятся в слоты `goal_1 ... goal_N`
* внутри одной темы слоты назначаются по порядку goal-колонок в исходном файле слева направо
* если в новых выгрузках по теме появляется дополнительная goal-колонка, ей назначается следующий свободный слот `goal_N+1`
* уже назначенные ранее goal-слоты внутри темы не должны перенумеровываться задним числом
* в численных данных они сохраняются как обычные метрики
* в proxy spreadsheet для каждой темы хранится человекочитаемая дешифровка этих слотов
* одинаковый номер цели между разными темами не означает одинаковый смысл

Финальный export-слой

Рабочий spreadsheet должен наполняться не из raw-слоя, а из export-view.

Export-view представляет собой wide-таблицу, где:

* dimensions выводятся фиксированным набором колонок
* metrics выводятся фиксированным набором колонок
* цели выводятся как `goal_1 ... goal_N`
* отсутствующие у конкретного клиента поля остаются пустыми

Именно этот слой используется как финальная рабочая таблица.

Выбор актуальной версии строки

Так как один и тот же отчетный grain может приходить из нескольких файлов, нужно хранить версии и выбирать актуальную.

Рекомендуемый подход:

* считать `row_hash` по фактическому набору dimensions строки
* если строка уже существует, выбирать актуальную версию по приоритету источника

Приоритет:

1. более поздняя `message_date`
2. более поздняя дата загрузки
3. технический tie-breaker по `file_id`

У актуальной строки:

* `is_current = true`

У вытесненных строк:

* `is_current = false`

Это позволяет:

* не терять историю
* не зависеть от “последнего письма”
* корректно обрабатывать повторные выгрузки и дозаливки

Технические требования к Apps Script

Apps Script должен:

* читать список тем из колонки A proxy spreadsheet
* по каждой теме искать письма за текущий месяц
* собирать все вложения
* считать или передавать на backend данные для дедупликации
* отправлять вложения в Supabase как есть
* не принимать решений о победившей версии файла
* не выполнять сложной бизнес-нормализации

Технические требования к backend

Backend должен:

* принимать сырые файлы
* классифицировать файл
* извлекать табличные данные
* извлекать дату отчета и при необходимости период
* нормализовать заголовки
* строить канонический sparse-слой
* собирать export wide-view
* публиковать актуальный слой в рабочий spreadsheet

Итог

Нужен pipeline, который:

1. берет из proxy spreadsheet список тем писем;
2. по каждой теме синхронизирует все вложения за текущий месяц;
3. дедуплицирует вложения по содержимому, а не по имени;
4. извлекает дату отчета из содержимого файла;
5. нормализует данные по заголовкам, а не по позиции столбцов;
6. поддерживает superset всех UTM, dimensions и метрик;
7. обезличивает цели в `goal_1 ... goal_N`;
8. использует proxy spreadsheet как справочник дешифровки целей по теме;
9. хранит внутренний канонический слой в устойчивой sparse-модели;
10. формирует финальную wide-витрину для рабочего spreadsheet только из актуальных строк.

Это решение должно быть устойчиво к:

* разному порядку столбцов
* отсутствию части полей
* разным наборам UTM
* разным наборам метрик
* хаотичным названиям файлов
* нескольким письмам и нескольким вложениям по одной теме
* повторным выгрузкам и частичным дозаливкам
